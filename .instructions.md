# Copilot Instructions for Idle Champions Code Redeemer Bot

## Project Overview

**Idle Champions Code Redeemer Bot** is a Discord bot that automatically scans for and redeems Idle Champions promo codes. It's a cloud-ready replacement for the browser extension with multi-user support, secure credential storage, and full-featured game interactions.

- **Task Runner**: ⚡ **Mise (MANDATORY)** - All commands MUST use `mise run`, never `bun run` or `npm`
- **Bot Framework**: discord.js 14.26+
- **Database**: SQLite3 (local, secure storage)
- **Language**: TypeScript 6.0+
- **Build System**: TypeScript compiler with watch mode
- **HTTP Client**: node-fetch 3.3+

## Critical Project Constraints

### 1. SSL Certificate Issue ⚠️

The Idle Champions API server has an **expired SSL certificate**. This is handled in two places:

- Set globally in `bot.ts` at startup: `process.env.NODE_TLS_REJECT_UNAUTHORIZED = '0'`
- Also configured in `idleChampionsApi.ts` via `https.Agent({ rejectUnauthorized: false })`

**Always remember**: This must be set BEFORE any HTTPS requests are made.

### 2. Instance ID Management ⚠️

When calling game APIs (redeem, open chests, blacksmith), **you must always**:

1. Fetch fresh user details via `idleChampionsApi.getUserDetails()`
2. Extract `instance_id` from the response (`details.instance_id`)
3. Pass it to the API call

Failing to do this causes "Outdated instance id" errors from the server. The `instance_id` changes with each user session/action.

### 3. API Query Parameter Format

All game API calls use **URL query parameters**, not JSON body:

```
POST https://server.url/~idledragons/post.php?call=redeemcoupon&user_id=X&hash=Y&instance_id=Z&code=ABC
```

The `idleChampionsApi` class handles this pattern consistently. When adding new API calls, follow this pattern.

### 4. Database Initialization

- Database location: `./data/idle.db` (configurable via `DB_PATH` env var)
- Foreign keys are enabled: `PRAGMA foreign_keys = ON`
- Tables created automatically on first run
- Credentials are stored plaintext (SQLite is local & secure)

- ✅ Docker builds use frozen lock files to guarantee identical production environments
- ✅ Lock file prevents silent dependency upgrades that could introduce bugs or vulnerabilities

### Schema

**users**: Stores Discord → Idle Champions credential mapping
**redeemed_codes**: Tracks code history with:

- `code` - The promo code
- `discord_id` - User who redeemed it
- `redeemed_at` - When redeemed
- `status` - success/expired/error
- `loot_detail` - JSON of rewards (gold, rubies, equipment)
- `is_public` - Whether code is shared (1) or private (0)
- `expires_at` - Expiration timestamp
  **pending_codes**: Queue of codes detected but not yet redeemed
  **audit_log**: Records all significant operations

## Release Signing & Attestation [OSPS-BR-06.01]

All official releases are cryptographically signed at build time using **Cosign (Keyless OIDC)** to ensure authenticity and integrity.

### What Gets Signed

1. **Docker Images** (during `.github/workflows/docker.yml`)
   - All images published to GHCR are signed
   - Signature stored in GHCR alongside image
   - Verifiable using `cosign verify`

2. **Release Attestations** (during `.github/workflows/release-attestation.yml`)
   - `attestation.json` - Complete metadata with hashes and build info
   - `attestation.sig` - Cosign signature over attestation
   - `attestation.crt` - Cosign certificate for verification
   - Automatically uploaded to GitHub Release
   - Automatically posted to release with verification instructions

### Signing Process

**Automatic**:

- Cosign keyless signing uses GitHub OIDC tokens
- No secrets stored; uses GitHub Actions authentication
- Works for both images and attestation files
- Fully auditable via GitHub Actions logs

**Workflow Integration**:


### Commit Message Format

```

<footer>
```

### Types

- **feat**: A new feature
- **fix**: A bug fix
- **docs**: Documentation only changes
- **style**: Changes that don't affect code meaning (formatting, missing semicolons, etc.)
- **refactor**: Code change that neither fixes a bug nor adds a feature
- **perf**: Code change that improves performance
- **test**: Adding or updating tests
- **chore**: Changes to build process, dependencies, or tooling
- **ci**: Changes to CI/CD configuration

### Scope

Scope is optional but recommended. Use the component/area of code:

- `api` - Game API client changes
- `bot` - Main Discord bot logic
- `database` - Database schema or queries
- `commands` - Discord slash commands
- `logging` - Logging system changes
- `ui` - User-facing messages or embeds

## Key Algorithms & Patterns

### Code Sharing System

### Public vs Private Codes

The bot supports automatic and manual code sharing between users:

**Private Codes** (Default)

- All codes start as private when redeemed via `/redeem`
- Only the user who redeemed it gets the rewards
- Not shared with other users unless explicitly made public
- **Auto-Public on Second User Success**: If a second user successfully redeems the same code, it automatically becomes public for both users

**Public Codes** (Explicit or Automatic)

- Made public automatically when a second user successfully redeems it
- Made public explicitly via `/makepublic` command
- Any user can redeem public codes via `/redeem` command
- If an explicitly public code becomes invalid on redemption, it's automatically switched back to private (may be a mistake)
- Only unexpired, successful codes show as public

### Workflow Example

1. User A redeems code `XYZ` via `/redeem code:XYZ` (stored as private)
2. User B redeems the same code `XYZ` via `/redeem code:XYZ` → code automatically becomes public
3. User A can see the code is now public in `/codes` command
4. User C can manually redeem via `/redeem code:XYZ` and will also see it as public
5. User A can explicitly make a code public via `/makepublic code:XYZ` (for exclusive codes they want to share)
6. If User D redeems an explicitly public code and it becomes invalid, it switches back to private automatically

### Database Fields

**redeemed_codes table fields:**

- `is_public` (integer) - 1 if public, 0 if private
- `expires_at` (datetime) - When code expires (null = never)
- `status` (text) - 'Success' or 'Code Expired' (only successful/expired redeems are stored)

### API Request Logging

**Purpose**: All HTTP requests/responses to the Idle Champions API are automatically logged for audit trail and debugging.

**Location**: `api-logs/` directory (auto-created on first request)

**File Format**: `{userIdShort}_{action}_{timestamp}.json`

- Example: `12345_redeemcoupon_2026-01-15T10-23-45-123Z.json`
- Each file contains request and response JSON

**Automatic Cleanup**:

- Files older than **24 hours** are automatically deleted
- Cleanup check runs every **1 hour**
- Triggered on bot startup via `apiRequestLogger.initialize()`

**Logged Actions**:

- `redeemcoupon` - Code redemption attempts
- `getuserdetails` - Account info retrieval
- `opengenericchest` - Chest openings
- `buysoftcurrencychest` - Chest purchases
- `useServerBuff` - Blacksmith upgrades
- `getserver` - Server info retrieval

**Sensitive Data Handling**:

- User IDs are truncated to first 5 characters in filenames
- Hash parameters are removed from logged URLs (replaced with `hash=***`)
- Code parameters are sanitized before logging

**Code Redemption History**

```sql
-- View all redeemed codes
SELECT code, discord_id, redeemed_at, status FROM redeemed_codes ORDER BY redeemed_at DESC;

-- View codes for specific user
SELECT code, redeemed_at, status, loot_detail FROM redeemed_codes
WHERE discord_id = '123456789'
ORDER BY redeemed_at DESC LIMIT 10;

-- Count redeemed codes per user
SELECT discord_id, COUNT(*) as code_count FROM redeemed_codes GROUP BY discord_id ORDER BY code_count DESC;

-- View public codes (available for sharing)
SELECT code, discord_id, redeemed_at, expires_at
FROM redeemed_codes
WHERE is_public = 1 AND (expires_at IS NULL OR expires_at > datetime('now'))
ORDER BY redeemed_at DESC;

-- View expired codes
SELECT code, discord_id, expires_at FROM redeemed_codes WHERE expires_at IS NOT NULL AND expires_at <= datetime('now');

-- Check if specific code exists
SELECT * FROM redeemed_codes WHERE code = 'ABC123';

-- Find codes with errors
SELECT code, discord_id, status FROM redeemed_codes WHERE status != 'success';
```

## Important Implementation Notes

### Async/Await Usage

- Always use async/await for promises (not .then())
- Always await database operations
- Always await Discord API calls
- Always await fetch requests

### Type Safety

- Use strict TypeScript settings
- Import types from discord.js and game type files
- Define interfaces for API options (see `idleChampionsApi.ts`)
- Avoid `any` type unless absolutely necessary

### Performance Considerations

- Bun startup time is much faster than Node.js
- Database queries are synchronous (sqlite3 limitation)
- API calls to game server can be slow (100-500ms)
- Code scanning happens on every message (keep regex efficient)

### Security Notes

- **User credentials stored plaintext in SQLite** (acceptable because it's local)
- Discord command options are stripped of sensitive data before logging
- Ephemeral messages used for credential display
- No credentials logged to console
