# Copilot Instructions for Idle Champions Code Redeemer Bot

## Project Overview

**Idle Champions Code Redeemer Bot** is a Discord bot that automatically scans for and redeems Idle Champions promo codes. It's a cloud-ready replacement for the browser extension with multi-user support, secure credential storage, and full-featured game interactions.

### Tech Stack
- **Runtime**: Bun 1.3.9+ (3-4x faster than Node.js) or Node.js 20+
- **Bot Framework**: discord.js 14.26+
- **Database**: SQLite3 (local, secure storage)
- **Language**: TypeScript 6.0+
- **Build System**: TypeScript compiler with watch mode
- **HTTP Client**: node-fetch 3.3+

## Critical Project Constraints

### 1. SSL Certificate Issue ⚠️
The Idle Champions API server has an **expired SSL certificate**. This is handled in two places:
- Set globally in `bot.ts` at startup: `process.env.NODE_TLS_REJECT_UNAUTHORIZED = '0'`
- Also configured in `idleChampionsApi.ts` via `https.Agent({ rejectUnauthorized: false })`

**Always remember**: This must be set BEFORE any HTTPS requests are made.

### 2. Instance ID Management ⚠️
When calling game APIs (redeem, open chests, blacksmith), **you must always**:
1. Fetch fresh user details via `idleChampionsApi.getUserDetails()`
2. Extract `instance_id` from the response (`details.instance_id`)
3. Pass it to the API call

Failing to do this causes "Outdated instance id" errors from the server. The `instance_id` changes with each user session/action.

### 3. API Query Parameter Format
All game API calls use **URL query parameters**, not JSON body:
```
POST https://server.url/~idledragons/post.php?call=redeemcoupon&user_id=X&hash=Y&instance_id=Z&code=ABC
```

The `idleChampionsApi` class handles this pattern consistently. When adding new API calls, follow this pattern.

### 4. Database Initialization
- Database location: `./data/idle.db` (configurable via `DB_PATH` env var)
- Foreign keys are enabled: `PRAGMA foreign_keys = ON`
- Tables created automatically on first run
- Credentials are stored plaintext (SQLite is local & secure)

## Code Style & Conventions

### TypeScript Configuration
- Target: ES2020+
- Module: CommonJS (for Node.js/Bun compatibility)
- Strict mode enabled
- All files must be type-safe

### Naming Conventions
- **Classes**: PascalCase (e.g., `IdleChampionsApi`, `Database`)
- **Functions**: camelCase (e.g., `executeCommand`, `scanMessageForCodes`)
- **Constants**: SCREAMING_SNAKE_CASE (e.g., `MAX_OPEN_CHESTS`, `DISCORD_TOKEN`)
- **Interfaces**: PascalCase with `I` prefix or no prefix (project uses no prefix, e.g., `CodeSubmitOptions`)

### File Organization
- One exported class or function per file
- Interfaces/types defined at top of file
- Imports grouped: node modules → project files → types
- Export: `export const/function/class` for top-level, `export default` for single exports

### Command Structure
All commands follow this pattern:
```typescript
import { SlashCommandBuilder, ChatInputCommandInteraction } from 'discord.js';

export const data = new SlashCommandBuilder()
  .setName('command_name')
  .setDescription('Description')
  .addStringOption(/* options */);

export async function execute(interaction: ChatInputCommandInteraction) {
  // Implementation
}
```

### Error Handling
- Use try-catch for async operations
- Log errors with context: `console.error('[CONTEXT] message:', error)`
- Return ephemeral error messages to Discord users
- Use `MessageFlags.Ephemeral` for sensitive responses

### Logging Convention
Use prefixed console logs for debugging:
```typescript
console.log('[SETUP] User credentials saved for:', interaction.user.id);
console.error('[API] Failed to fetch user details:', error);
```

Prefix format: `[COMPONENT_NAME]` (e.g., `[BOT]`, `[SETUP]`, `[API]`, `[DB]`)

## Command Development

### Creating a New Command
1. Create file in `src/bot/commands/yourcommand.ts`
2. Use `SlashCommandBuilder` to define parameters
3. Implement `execute(interaction)` function
4. Export both `data` and `execute`
5. Commands are auto-loaded by `bot.ts`

### Common Patterns

**Deferring Replies** (for long operations):
```typescript
await interaction.deferReply({ flags: MessageFlags.Ephemeral });
// Do work...
await interaction.editReply({ content: 'Done!' });
```

**Getting User Credentials**:
```typescript
const user = await userManager.getCredentials(interaction.user.id);
if (!user) {
  await interaction.reply({ content: 'Please run /setup first', flags: MessageFlags.Ephemeral });
  return;
}
```

**Calling Game APIs**:
```typescript
// 1. Get fresh user details for instance_id
const details = await idleChampionsApi.getUserDetails({
  server: user.server,
  user_id: user.user_id,
  hash: user.user_hash,
});

// 2. Use instance_id from response
const result = await idleChampionsApi.redeemCode({
  server: user.server,
  code,
  user_id: user.user_id,
  hash: user.user_hash,
  instanceId: details.instance_id,
});
```


### Schema
**users**: Stores Discord → Idle Champions credential mapping
**redeemed_codes**: Tracks code history with loot details
**pending_codes**: Queue of codes detected but not yet redeemed
**audit_log**: Records all significant operations

## Key Algorithms & Patterns

### Code Scanning (Auto-detection)
In `codeScanner.ts`, codes are detected via regex pattern. Current pattern matches Idle Champions promo codes in messages. When updating:
- Test regex thoroughly with various code formats
- Avoid false positives
- Log detections for auditing

### Discord Event Handling
Main event listeners in `bot.ts`:
- `ready`: Bot connected
- `interactionCreate`: Slash commands & buttons
- `messageCreate`: Code auto-scanning

### API Response Pattern
All `idleChampionsApi` methods:
1. Build URL with query parameters
2. Fetch with HTTPS agent (ignores cert)
3. Parse JSON response
4. Validate response structure
5. Return typed result or throw error

## Testing & Debugging

### Debug Logging
- Use `debugLogger.initDebugLogger()` to log API responses
- Logs saved to `debug/` folder with auto-cleanup (keeps last 10)
- Useful for troubleshooting API issues

### Common Issues

| Issue | Solution |
|-------|----------|
| "Outdated instance id" error | Fetch fresh user details, extract instance_id, use it immediately |
| SSL certificate errors | Ensure `NODE_TLS_REJECT_UNAUTHORIZED=0` is set at startup |
| Command not appearing in Discord | Restart bot, check guild_id in .env, reload Discord client |
| Database locked | Close all other connections, check for file locks |
| Code not detecting | Check regex pattern in `codeScanner.ts`, test against real codes |

### Debugging Tips
1. Check console logs with `[COMPONENT]` prefix
2. Enable debug logger to see API responses
3. Use Discord channel history to test code detection
4. Check SQLite with `sqlite3 data/idle.db` in terminal
5. Verify environment variables are loaded: `console.log(process.env)`

## Important Implementation Notes

### Async/Await Usage
- Always use async/await for promises (not .then())
- Always await database operations
- Always await Discord API calls
- Always await fetch requests

### Type Safety
- Use strict TypeScript settings
- Import types from discord.js and game type files
- Define interfaces for API options (see `idleChampionsApi.ts`)
- Avoid `any` type unless absolutely necessary

### Performance Considerations
- Bun startup time is much faster than Node.js
- Database queries are synchronous (sqlite3 limitation)
- API calls to game server can be slow (100-500ms)
- Code scanning happens on every message (keep regex efficient)

### Security Notes
- **User credentials stored plaintext in SQLite** (acceptable because it's local)
- Discord command options are stripped of sensitive data before logging
- Ephemeral messages used for credential display
- No credentials logged to console
