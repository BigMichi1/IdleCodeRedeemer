# Copilot Instructions for Idle Champions Code Redeemer Bot

## Project Overview

**Idle Champions Code Redeemer Bot** is a Discord bot that automatically scans for and redeems Idle Champions promo codes. It's a cloud-ready replacement for the browser extension with multi-user support, secure credential storage, and full-featured game interactions.

- **Language**: TypeScript 6.0+
- **Build System**: TypeScript compiler with watch mode

## Critical Project Constraints

### 1. SSL Certificate Issue ⚠️

The Idle Champions API server has an **expired SSL certificate**. This is handled in two places:

- Set globally in `bot.ts` at startup: `process.env.NODE_TLS_REJECT_UNAUTHORIZED = '0'`
- Also configured in `idleChampionsApi.ts` via `https.Agent({ rejectUnauthorized: false })`

**Always remember**: This must be set BEFORE any HTTPS requests are made.

### 2. Instance ID Management ⚠️

When calling game APIs (redeem, open chests, blacksmith), **you must always**:

1. Fetch fresh user details via `idleChampionsApi.getUserDetails()`
2. Extract `instance_id` from the response (`details.instance_id`)
3. Pass it to the API call

Failing to do this causes "Outdated instance id" errors from the server. The `instance_id` changes with each user session/action.

### 3. API Query Parameter Format

All game API calls use **URL query parameters**, not JSON body:

```
POST https://server.url/~idledragons/post.php?call=redeemcoupon&user_id=X&hash=Y&instance_id=Z&code=ABC
```

The `idleChampionsApi` class handles this pattern consistently. When adding new API calls, follow this pattern.

### 4. Database Initialization

- Database location: `./data/idle.db` (configurable via `DB_PATH` env var)
- Foreign keys are enabled: `PRAGMA foreign_keys = ON`
- Tables created automatically on first run
- Credentials are stored plaintext (SQLite is local & secure)

## Key Algorithms & Patterns

### Code Sharing System

### Public vs Private Codes

The bot now supports sharing codes between users:

**Private Codes** (Default)

- Redeemed via `/redeem` command
- Only the user who redeemed it gets the rewards
- Not shared with other users
- Stored in database for user's history

**Public Codes**

- Made public via `/makepublic` command
- Any user can redeem via `/redeempublic`
- Only unexpired, successful codes can be made public
- When new users run `/setup`, public codes are automatically redeemed for them

### Workflow Example

1. User A redeems code `XYZ` via `/redeem code:XYZ` (private)
2. User A likes the code and runs `/makepublic code:XYZ` to share it
3. User B joins and runs `/setup` → automatically redeems `XYZ` and gets rewards
4. User C can manually redeem via `/redeempublic code:XYZ` to see and redeem the shared code
5. User D can run `/redeempublic` (no code) to see all available public codes

### Expiration Tracking

Codes can expire on the game server:

- When a code fails to redeem with "expired" error message, bot marks it as `expired`
- Expired codes no longer appear in public code lists
- Each code tracks when it expires via `expires_at` timestamp

### Database Fields

**redeemed_codes table additions:**

- `is_public` (integer) - 1 if public, 0 if private
- `expires_at` (datetime) - When code expires (null = never)

## Testing & Debugging

### Debug Logging

- Use `debugLogger.initDebugLogger()` to log API responses
- Logs saved to `debug/` folder with auto-cleanup (keeps last 10)
- Useful for troubleshooting API issues
- API request logs in `api-logs/` provide comprehensive audit trail

### Common Issues

| Issue                            | Solution                                                          |
| -------------------------------- | ----------------------------------------------------------------- |
| "Outdated instance id" error     | Fetch fresh user details, extract instance_id, use it immediately |
| SSL certificate errors           | Ensure `NODE_TLS_REJECT_UNAUTHORIZED=0` is set at startup         |
| Command not appearing in Discord | Restart bot, check guild_id in .env, reload Discord client        |
| Database locked                  | Close all other connections, check for file locks                 |
| Code not detecting               | Check regex pattern in `codeScanner.ts`, test against real codes  |

### Debugging Tips

1. Check console logs with `[COMPONENT]` prefix
2. Enable debug logger to see API responses
3. Use Discord channel history to test code detection
4. Check SQLite with `sqlite3 data/idle.db` in terminal
5. Verify environment variables are loaded: `console.log(process.env)`

## Important Implementation Notes

### Async/Await Usage

- Always use async/await for promises (not .then())
- Always await database operations
- Always await Discord API calls
- Always await fetch requests

### Type Safety

- Use strict TypeScript settings
- Import types from discord.js and game type files
- Define interfaces for API options (see `idleChampionsApi.ts`)
- Avoid `any` type unless absolutely necessary

### Performance Considerations

- Bun startup time is much faster than Node.js
- Database queries are synchronous (sqlite3 limitation)
- API calls to game server can be slow (100-500ms)
- Code scanning happens on every message (keep regex efficient)

### Security Notes

- **User credentials stored plaintext in SQLite** (acceptable because it's local)
- Discord command options are stripped of sensitive data before logging
- Ephemeral messages used for credential display
- No credentials logged to console
